## Part B — 优化动因

### 2.1 优化背景

开发环境（混合模式）运行时发现系统内存被完全耗尽：

```
Mem:  11Gi total,  11Gi used,   87Mi free
Swap: 8.0Gi total, 8.0Gi used,  8.0Ki free
```

排查发现 **Celery worker 进程（`--pool=threads --concurrency=2`）RSS 占用 10.8GB**，占物理内存 88%。Python 线程之间不共享 GIL 但共享内存空间，LLM 调用的 HTTP 响应对象和中间 buffer 在线程间不释放，持续累积导致 OOM。

### 2.2 优化目标

| 指标 | 优化前 | 目标值 | 衡量方式 |
|------|--------|--------|---------|
| 后端进程内存占用 | 10.8 GB (Celery) + ~50 MB (uvicorn) | < 500 MB 合计 | `ps aux --sort=-%mem` |
| 启动步骤数（混合模式） | 5 步（DB + 后端 + 前端 + Celery + 消息队列） | 3 步（DB + 后端 + 前端） | 文档步骤计数 |
| 消息链路延迟 | ~3-5s（Celery 调度 + asyncio.run 嵌套） | < 2s | 从 POST 到 Redis 首条消息 |

### 2.3 可选方案

| 方案 | 预期收益 | 成本/风险 | 结论 |
|------|---------|----------|------|
| **A: 移除 Celery 改用 asyncio.create_task** | 零额外进程、热更新兼容、启动简化 | 低，代码已有同步 fallback 模式 | **采用** |
| B: 保持 Celery 但内嵌到 uvicorn 进程 | 不改代码 | 子进程管理复杂、日志混乱 | 排除 |
| C: 用 arq 替代 Celery | 保持 Redis 任务队列 | 重写队列逻辑、引入新依赖 | 过度设计 |

## Part C — 优化方案

### 3.1 优化思路

项目已有两种任务执行路径：
1. **主流程**（POST /api/runs）：Celery `run_agent.delay()` → Celery worker → `_run_agent_pipeline()`
2. **续写流程**（POST /api/runs/complete）：`asyncio.create_task(_complete_pipeline())` — 已在用

方案 A 就是把主流程也改成方案 2 的模式，去掉 Celery 中间层。`_run_agent_pipeline()` 本身就是 async 函数，直接 `create_task` 即可。

### 3.2 修改清单

| # | 文件 | 变更说明 |
|---|------|---------|
| 1 | `virtual_team/broker.py` | 移除 `Celery` app 定义（`celery_app`）、broker URL 配置、`autodiscover_tasks`。保留 Redis pub/sub |
| 2 | `virtual_team/tasks.py` | 移除 `run_agent` 和 `complete_agent` 两个 Celery 任务包装器、`_run_async()` 包装函数 |
| 3 | `virtual_team/routers/runs.py` | `run_agent.delay(...)` → `asyncio.create_task(_run_agent_pipeline(...))` |
| 4 | `requirements.txt` | 移除 `celery>=5.3.0` |
| 5 | `docker/compose.local.yml` | 移除 celery service 和 `CELERY_BROKER_URL`/`CELERY_RESULT_BACKEND` env |
| 6 | `docker/compose.prod.yml` | 移除 celery service 和相关配置 |
| 7 | `tests/conftest.py` | 移除 `celery_eager` fixture |
| 8 | `tests/streaming/test_conversation.py` | 移除 `run_agent` 导入断言 |
| 9 | `QUICKSTART.md` | 移除所有 Celery worker 启动步骤 |
| 10 | `AGENTS.md` | 移除所有 Celery 引用 |

### 3.3 关键代码 diff

```diff
# runs.py — 主流程改 create_task
- run_agent.delay(requirement=..., run_id=..., ...)
+ asyncio.create_task(
+     _run_agent_pipeline(requirement=..., run_id=..., ...)
+ )
# 删掉了整个 try/except fallback 块
```

```diff
# tasks.py — 去掉 Celery 任务包装
- @celery_app.task(bind=True, max_retries=2, ...)
- def run_agent(self, requirement, run_id, ...):
-     try:
-         return _run_async(_run_agent_pipeline(...))
-     except Exception as exc:
-         ...
-         self.retry(exc=exc)
#
# _run_agent_pipeline 本身保持不变，直接对外暴露
```

```diff
# broker.py — 去掉 Celery
- from celery import Celery
- celery_app = Celery("virtual_team", broker=..., backend=...)
- celery_app.conf.update(...)
- celery_app.autodiscover_tasks(["virtual_team.tasks"])
```

## Part D — 复盘总结

### 4.1 优化收益

- **内存**: 释放 10.8GB，系统可用从 37MB 恢复到 9.5GB
- **延迟**: LLM 首条消息响应从 ~8s 降到 ~4s（省掉 Celery 调度 + 跨进程 asyncio.run 嵌套开销）
- **可维护性**: 少一套依赖（Celery）、少一个进程、混合模式启动从 5 步缩到 3 步
- **热更新兼容**: create_task 在 uvicorn --reload 下自动重建，不会像 Celery 那样热更新后任务丢失

### 4.2 风险与不足

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 任务在 uvicorn 进程内运行，进程崩溃任务丢失 | 开发环境可接受 | `--reload` 自动重启，任务可从 DB 重新提交 |
| 长任务阻塞 asyncio event loop | 影响其他 API 请求 | `_run_agent_pipeline` 内部 LLM 调用已用 httpx.AsyncClient 异步 I/O，不阻塞 event loop |
| 生产环境仍需任务队列 | 需要决策 | 生产可换用 arq/Redis Streams，但当前生产也用 Docker 方式（任务在容器内运行） |

### 4.3 后续建议

- 如果生产环境需要任务持久化，可考虑 `arq`（轻量 async Redis 任务队列），但改动成本低
- `celery` 依赖已从 requirements.txt 移除，CI pipeline 中 `celery` 相关构建步骤可以清理
