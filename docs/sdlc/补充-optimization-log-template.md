# 补充模板：优化日志模板

> **补充说明**：本模板填补 SDLC 文档体系中技术优化日志模板的空白。与 `补充-fix-log-template.md`（修复日志）的区别在于：修复日志记录的是**缺陷根因分析**（东西坏了，为什么会坏），本模板记录的是**主动改进**（没坏，但可以更好）。
>
> 适用场景：性能优化、重构改进、架构升级、依赖升级、技术债务清理、CI/CD 流水线优化。

---

## 1. 模板说明

优化日志应在优化完成后编写，包含以下四个部分：

| Part | 内容 | 目的 |
|------|------|------|
| A | 基本信息 | 快速定位：时间、模块、优化类型 |
| B | 优化动因 | 解释：为什么做、怎么衡量的 |
| C | 优化方案 | 记录：做了什么改动、前后对比 |
| D | 复盘总结 | 沉淀：收益、风险、后续建议 |

---

## Part A — 基本信息

| 字段 | 说明 | 填写示例 |
|------|------|---------|
| 标题 | 简短描述优化内容 | Vite 构建从 45s 优化至 12s |
| 优化编号 | 唯一标识，格式：OPT-YYYYMMDD-XXX | OPT-20260630-001 |
| 日期 | 优化完成日期 | 2026-06-30 |
| 作者 | 优化实施人员 | Sisyphus |
| 涉及模块 | 涉及的模块、服务、文件 | `frontend/vite.config.ts`、`frontend/src/` |
| 优化类型 | 性能 / 重构 / 安全 / 体验 / 可维护性 / 成本 / 其他 | 性能 |
| 触发方式 | 主动规划 / 技术债务 / 线上监控告警 / 代码审查 / 团队倡议 | 主动规划 |
| 评审人 | 代码审查人员（可选） | — |

## Part B — 优化动因

### 2.1 优化背景

**为什么要做这个优化**：

> 示例：随着项目模块增加，Vite 开发服务器冷启动时间从 8s 增长到 45s，团队成员每日累计等待超过 10 分钟，影响开发效率。

### 2.2 优化目标

**可量化的目标**：

| 指标 | 优化前 | 目标值 | 衡量方式 |
|------|--------|--------|---------|
| Vite 冷启动时间 | 45s | < 15s | `time npm run dev` |
| 生产构建体积 | 2.3 MB | < 1.5 MB | `npm run build && du -sh dist/` |
| Lighthouse Performance | 72 | > 85 | Lighthouse CI |

### 2.3 可选方案（如有多种方案比较）

| 方案 | 预期收益 | 成本/风险 | 结论 |
|------|---------|----------|------|
| A: 接入 lazy loading | 减少首屏 JS 30% | 中等，需重构路由 | 推荐 |
| B: 升级 Vite 6 | 构建快 20% | 低，版本升级 | 备选 |
| C: 不做 | — | 持续累积技术债 | 排除 |

---

## Part C — 优化方案

### 3.1 优化思路

**整体策略**：

> 示例：利用 Vite 6 的 Rolldown 打包 + 手动 Chunk 拆分 + 组件懒加载，减少首屏加载体积。

### 3.2 修改清单

| # | 文件 | 变更说明 |
|---|------|---------|
| 1 | `vite.config.ts` | 启用 Rolldown，配置 `manualChunks` 将 `react`、`utils` 拆为独立 vendor |
| 2 | `src/router/index.tsx` | 路由组件改为 `React.lazy()` + `Suspense` |
| 3 | `src/components/HeavyTable.tsx` | 提取为异步加载的 `ChunkHeavyTable` |

### 3.3 关键代码 diff（可选）

```diff
// vite.config.ts
export default defineConfig({
  build: {
+   rollupOptions: {
+     output: {
+       manualChunks: {
+         vendor: ['react', 'react-dom', 'react-router-dom'],
+         utils: ['axios', 'zustand', 'crypto-js'],
+       },
+     },
+   },
  },
});
```

### 3.4 优化结果

**前后对比**：

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| Vite 冷启动时间 | 45s | 12s | **73% ↓** |
| 生产构建体积 | 2.3 MB | 1.2 MB | **48% ↓** |
| Lighthouse Performance | 72 | 91 | **+19** |

### 3.5 回滚方案（可选）

| 场景 | 操作 | 预期耗时 |
|------|------|---------|
| 优化导致兼容性问题 | 回退 `vite.config.ts` 至上一版本 | 5 分钟 |

## Part D — 复盘总结

### 4.1 收益总结

| # | 收益 | 量化 |
|---|------|------|
| 1 | 开发效率提升 | 每人每天节省约 5 分钟等待时间 |
| 2 | 用户体验改善 | 首屏加载时间从 3.2s 降至 1.1s |
| 3 | CI 构建加速 | PR 流水线从 4 分钟缩短至 2 分钟 |

### 4.2 风险与权衡

| # | 风险/代价 | 说明 |
|---|----------|------|
| 1 | 部分组件懒加载增加了首次交互延迟 | 按需加载的组件首次点击有 200ms 加载延迟，可接受 |
| 2 | 构建配置复杂度提升 | 需要维护 `manualChunks` 策略，随依赖变化调整 |

### 4.3 后续建议

| # | 建议 | 优先级 |
|---|------|--------|
| 1 | 在 CI 中加入构建体积监控，超阈值告警 | P1 |
| 2 | 定期检查 `manualChunks` 策略是否仍然合理（每季度） | P2 |
| 3 | 接入 Lighthouse CI 自动门禁 | P2 |

## 2. 文件命名规范

格式：`opt-log-YYYY-MM-DD-简短描述.md`

| 部分 | 规则 | 示例 |
|------|------|------|
| 前缀 | 统一以 `opt-log-` 开头 | `opt-log-` |
| 日期 | `YYYY-MM-DD` 格式 | `2026-06-30` |
| 描述 | 简短 kebab-case，3-5 个英文单词概括优化内容 | `vite-build-optimization` |
| 后缀 | `.md` | `.md` |

| 示例 | 说明 |
|------|------|
| `opt-log-2026-06-30-vite-build-optimization.md` | Vite 构建优化 |
| `opt-log-2026-07-15-lighthouse-performance.md` | 前端性能优化 |
