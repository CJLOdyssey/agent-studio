## 2. 任务概述

### 2.1 测试目标

1. **功能验证**：验证系统各功能模块的实现与需求规格说明书保持一致。
2. **接口验证**：验证后端 API 接口符合设计规范。
3. **业务流程验证**：验证关键端到端业务流程的正确闭环。
4. **兼容性验证**：验证前端应用在不同浏览器及视口尺寸下的渲染和交互表现正常。
5. **稳定性验证**：验证系统在并发请求、长时间运行及异常输入条件下的稳定性。

### 2.2 测试范围

**纳入测试范围**：[列举纳入测试的功能模块和技术层次，按测试类型分组描述。例如：前端应用的单元测试与组件测试；后端接口的功能测试与集成测试；前后端联调及关键业务流程的端到端测试；数据库层的数据操作正确性验证；实时通信链路的正确性和稳定性；认证与授权模块的权限控制] 

**不纳入测试范围**：[列举明确不纳入测试的范围，并说明理由。例如：第三方服务的内部逻辑；基础设施部署脚本；第三方依赖库的内部逻辑]

### 2.3 测试策略

采用分层测试策略，按照测试金字塔模型分为四个层次：

1. **单元测试层**：开发工程师同步编写，覆盖前端组件、工具函数及后端业务逻辑函数。使用的框架由项目技术栈决定（如 Vitest、Jest、JUnit、pytest 等）。
2. **集成测试层**：验证模块间接口交互及数据流转，重点覆盖 API 请求响应链、数据访问层与数据库交互、中间件及编排引擎的执行逻辑。
3. **系统测试层**：在完整部署环境中验证系统整体功能，包括端到端业务流程、并发场景、性能基准及安全合规。
4. **验收测试层**：由产品负责人或业务方执行，验证系统是否满足业务需求及上线标准。

## 3. 测试类型与分级

### 3.1 单元测试

**目标**：验证最小可测试单元的正确性。

**范围**：[前端组件渲染、自定义 Hook 逻辑、工具函数、状态管理；后端业务逻辑函数、数据访问层查询函数等]

**执行标准**：[语句覆盖率不低于 X%，分支覆盖率不低于 X%，函数覆盖率不低于 X%，行覆盖率不低于 X%]。新增功能模块必须配套对应的单元测试用例，并在代码提交前本地执行通过。

**工具**：[前端测试框架]、[后端测试框架]

### 3.2 集成测试

**目标**：验证多个模块协同工作时的数据流转和接口兼容性。

**范围**：[API 接口测试（CRUD、参数校验、权限控制）；数据库集成测试（ORM 与数据操作一致性）；消息队列或事件总线测试（发布/订阅及任务调度）；实时通信测试（WebSocket 连接建立、消息推送及断线重连）]

**执行标准**：所有对外暴露的 API 接口必须有对应的集成测试覆盖。数据库测试使用内存数据库或测试容器 + fixture 隔离，确保测试间无状态污染。涉及外部服务的测试使用 mock 或 stub 模式。

### 3.3 系统测试

**目标**：在完整的系统环境中验证功能、性能、兼容性、安全性。

**范围**：功能系统测试（10 大模块全部功能点）；业务流程测试（团队创建 → Agent 配置 → 工具绑定 → 项目运行 → 结果查看）；并发测试；兼容性测试（Chrome、Firefox、Edge、Safari 最新两个主版本）；安全性测试（SQL 注入、XSS、越权访问、敏感信息泄露）。

### 3.4 验收测试

**目标**：验证系统是否满足业务需求及上线条件。

**范围**：核心业务流程的端到端验收；需求规格说明书中标记为高优先级的功能点；用户界面的易用性和一致性检查。

**执行方**：产品负责人、测试负责人、技术负责人联合执行。

### 3.5 冒烟测试

**目标**：在每个测试迭代开始前，快速验证被测版本的基础功能可用。

**内容**：前端应用可正常编译并启动；后端服务可正常启动并响应健康检查接口；数据库连接正常，基础 CRUD 接口可访问；登录/认证流程可用。

**通过标准**：冒烟测试用例 100% 通过。任何阻塞性失败将直接退回开发修复。

### 3.6 回归测试

**目标**：在缺陷修复、功能新增或配置变更后，验证已有功能未受影响。

**策略**：每次迭代结束前执行全量回归测试；缺陷修复后，针对修复点及其关联模块执行定向回归；使用自动化测试套件覆盖高频回归场景；回归测试范围根据代码变更影响分析动态调整。

## 6. 测试用例设计方法

### 6.1 等价类划分法

将输入域划分为若干等价类，从每个等价类中选取代表性数据作为测试输入，以用较少的用例覆盖较大的输入空间。

**应用场景**：表单输入字段的校验（如 Agent 名称长度、API 密钥格式）、分页查询参数、枚举类型参数。

**示例**：Agent 名称输入字段，需求限制为 1-64 个字符。
- 有效等价类：1 个字符、64 个字符、32 个字符的中英文混合
- 无效等价类：空字符串、65 个字符、特殊符号（如 `< > &`）、纯空格

### 6.2 边界值分析法

针对输入域的边界条件设计测试用例，因为大量缺陷往往出现在边界附近。

**应用场景**：数值范围校验（如团队最大成员数、会话最大消息数）、字符串长度限制（如提示词模板的最大长度）、数组/列表的元素数量边界。

**示例**：团队最大成员数限制为 10 人。
- 边界值：0 人（空团队）、1 人、10 人、11 人（超限）
- 结合等价类，补充正常值：5 人

### 6.3 场景法

基于业务流程和用户场景设计测试用例，覆盖基本流、备选流及异常流。

**应用场景**：端到端业务流程（创建团队 → 配置 Agent → 运行项目 → 查看结果）、状态流转测试（会话的创建、运行中、已完成、失败状态）、并发场景（多用户同时编辑同一资源）。

**示例**：项目运行场景
- 基本流：创建会话 → 输入需求 → 选择团队 → 提交运行 → 等待完成 → 查看结果
- 备选流 1：运行过程中点击"继续生成"，验证续接流程
- 备选流 2：运行过程中取消任务，验证状态回滚
- 异常流：运行过程中后端服务重启，验证断线恢复及数据一致性

## 7. 缺陷管理

### 7.1 缺陷分类

| 分类维度 | 类别 | 说明 |
|----------|------|------|
| 按性质 | 功能缺陷 | 功能实现与需求不符或缺失 |
| | 接口缺陷 | API 请求/响应格式、状态码、参数校验错误 |
| | 性能缺陷 | 响应时间超时、内存泄漏、CPU 占用过高 |
| | 兼容性缺陷 | 特定浏览器或设备上的显示或功能异常 |
| | 安全缺陷 | 越权访问、数据泄露、注入攻击 |
| | UI 缺陷 | 界面布局错乱、样式不一致、交互体验差 |
| 按引入阶段 | 需求缺陷 | 需求描述模糊、矛盾或遗漏 |
| | 设计缺陷 | 架构设计或接口设计不合理 |
| | 编码缺陷 | 代码逻辑错误、边界处理缺失 |
| | 配置缺陷 | 环境配置、依赖版本不匹配 |

### 7.2 缺陷严重等级

| 等级 | 定义 | 响应时限 | 示例 |
|------|------|----------|------|
| 致命 (P0) | 导致系统崩溃、数据丢失或核心功能完全不可用 | 2 小时内 | 后端服务启动失败、数据库连接池耗尽、关键 API 500 错误 |
| 严重 (P1) | 核心功能受损或存在重大安全漏洞 | 24 小时内 | 认证绕过、项目运行无响应、WebSocket 连接无法建立 |
| 一般 (P2) | 非核心功能异常或体验明显下降 | 3 个工作日内 | 分页数据排序错误、提示信息不准确、界面轻微错位 |
| 轻微 (P3) | 不影响功能的细节问题或优化建议 | 下个迭代 | 日志格式不规范、文案拼写错误、未使用的导入警告 |

### 7.3 缺陷处理流程

1. **发现与提交**：测试人员发现缺陷后，在缺陷管理系统中提交，包含复现步骤、实际结果、预期结果、环境信息及截图/日志附件。
2. **确认与分配**：测试负责人或开发负责人对缺陷进行确认，判定是否为有效缺陷，并分配给对应开发人员。
3. **修复与验证**：开发人员修复缺陷后，提交代码并关联缺陷单。测试人员验证修复结果，确认通过后关闭缺陷。
4. **拒绝与申诉**：若开发人员判定为无效缺陷或无法复现，可标记为"拒绝"或"需要更多信息"，测试人员可补充信息后重新打开。
5. **关闭与归档**：验证通过的缺陷关闭归档；延期处理的缺陷需经项目经理审批并记录风险。

## 8. 通过准则

### 8.1 测试准入条件

- 被测版本已通过编译/构建，无构建错误
- 冒烟测试用例 100% 通过
- 单元测试无新增失败（已有 2 个已知失败需记录在案并持续跟踪）
- 测试环境部署完成且与生产环境配置一致

### 8.2 测试通过标准

| 指标 | 通过标准 | 说明 |
|------|----------|------|
| 冒烟测试通过率 | 100% | 所有冒烟用例必须全部通过 |
| 单元测试覆盖率 | [语句 >= X%，分支 >= X%，函数 >= X%，行 >= X%] | 以覆盖率工具报告为准 |
| 集成测试用例通过率 | >= 95% | 允许存在未阻塞的轻微缺陷 |
| 系统测试用例通过率 | >= 95% | 核心业务流必须 100% 通过 |
| 验收测试用例通过率 | 100% | 验收用例全部通过方可上线 |
| 致命缺陷遗留数 | 0 | 任何 P0 缺陷必须修复后方可上线 |
| 严重缺陷遗留数 | 0 | 任何 P1 缺陷必须修复后方可上线 |
| 一般缺陷遗留数 | <= 5 | 遗留 P2 缺陷需经评估不影响核心功能 |
| 缺陷修复验证率 | 100% | 所有已修复缺陷必须经过验证确认 |

### 8.3 上线风险评估

若存在以下任一情况，需启动上线风险评估会议，由项目经理、技术负责人、测试负责人共同决策：

- 致命或严重缺陷未修复完毕
- 核心业务流测试用例未全部通过
- 性能指标未达到预设基线
- 安全测试发现高危漏洞未修复
- 回归测试中发现新的阻塞性问题

## 4. 用例维护规范

1. **新增用例**：每次需求变更或缺陷修复后，需评估是否需要补充新的测试用例。新增用例编号遵循原有编号规则顺序递增。
2. **用例评审**：测试用例在编写完成后需由测试负责人组织评审，确保覆盖度、可执行性及预期结果的明确性。
3. **用例更新**：当功能需求发生变更时，需同步更新对应的测试用例，并标记版本号。
4. **用例归档**：已废弃功能对应的测试用例需标记为"废弃"状态，保留历史记录但不纳入执行范围。
5. **自动化关联**：适合自动化的用例（尤其是回归测试用例）需逐步转化为自动化脚本，并在用例表中标注对应的脚本路径。
