## 2. 通用编程原则

### 2.1 KISS (Keep It Simple and Stupid)

代码应当简单直接，避免过度设计。每个函数、类、模块都应保持最小复杂度。

### 2.2 DRY (Don't Repeat Yourself)

禁止重复代码。相同的逻辑应当抽取为公共函数或工具类，通过参数化实现复用。

每个模块、类、函数只负责一项明确的职责。当发现职责过多时，应当拆分。

## 5. 注释规范

### 5.1 注释原则

- 注释应当解释 "为什么" 而非 "做什么"
- 代码本身应当清晰自解释，避免无意义的注释
- 注释需与代码保持同步，及时更新

### 5.2 文件/模块注释模板

```
/**
 * [占位符: 文件功能描述]
 *
 * @author [占位符: 作者]
 * @date [占位符: 创建日期]
 * @version [占位符: 版本号]
 * @copyright [占位符: 版权信息]
 */
```

### 5.3 类/接口注释模板

# 第一部分：代码仓库规范

## 1. 仓库结构

### 1.1 仓库划分策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| [单仓库 Monorepo] | [所有代码在同一仓库，按目录划分模块] | [中小团队/前后端一体化/组件库] |
| [多仓库 Multi-repo] | [前后端/各服务独立仓库] | [大团队/微服务/独立部署单元] |
| [混合模式] | [核心代码统一，外围服务独立] | [中大型项目] |

### 1.2 仓库命名规范

```
[组织前缀]-[项目名称]-[模块名称]
示例：myteam-order-service、myteam-frontend-web
```

### 1.3 目录结构规范

# 第二部分：分支管理规范

## 2. 分支模型

### 2.1 分支类型与说明

| 分支类型 | 名称格式 | 来源 | 合并目标 | 生命周期 | 对应环境 |
|---------|---------|------|---------|---------|---------|
| **主分支** | `main` / `master` | — | — | 永久 | 生产(Production) |
| **开发分支** | `develop` | `main` | `main` | 永久 | 测试(Test) |
| **特性分支** | `feature/[需求ID]-[简述]` | `develop` | `develop` | 需求完成即删除 | 开发(Dev) |
| **发布分支** | `release/[版本号]` | `develop` | `develop` + `main` | 发布完成即删除 | 预发(Staging) |
| **热修复分支** | `hotfix/[问题ID]-[简述]` | `main` | `main` + `develop` | 修复完成即删除 | — |
| **BUG修复分支** | `bugfix/[问题ID]-[简述]` | `develop` | `develop` | 修复完成即删除 | 开发(Dev) |

> 也可采用 **Trunk-based**（主干开发）模型：所有分支从 `main` 拉出，高频合并回 `main`，适用于 CI/CD 成熟度高的团队。

### 2.2 分支模型选择

| 因素 | Git-flow（推荐） | Trunk-based |
|------|-----------------|-------------|
| 团队规模 | [5 人以上] | [5 人以下] |
| 发布频率 | [按版本发布] | [每日/持续发布] |
| CI/CD 成熟度 | [中等] | [高] |
| 环境数量 | [多环境(dev/test/staging/prod)] | [精简环境] |

## 3. 分支操作规范

### 3.1 创建分支

```bash
# 特性分支
git checkout develop
git pull
git checkout -b feature/PROJ-123-add-login-page

# 发布分支
git checkout develop
git pull
git checkout -b release/v1.2.0

# 热修复分支
git checkout main
git pull
git checkout -b hotfix/BUG-456-fix-npe
```

### 3.2 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 特性分支 | `feature/[需求编号]-[kebab-case描述]` | `feature/PROJ-123-add-login-page` |
| 发布分支 | `release/v[主版本].[次版本].[修订号]` | `release/v1.2.0` |
| 热修复分支 | `hotfix/[缺陷编号]-[kebab-case描述]` | `hotfix/BUG-456-fix-npe` |
| 缺陷修复 | `bugfix/[缺陷编号]-[kebab-case描述]` | `bugfix/BUG-457-fix-typo` |

### 3.3 合并策略

| 场景 | 策略 | 命令 |
|------|------|------|
| feature → develop | Merge（保持提交历史） | `git merge --no-ff feature/xxx` |
| release → main | Merge（打 Tag） | `git merge --no-ff release/v1.0.0` |
| release → develop | Merge（回合修复） | `git merge release/v1.0.0` |
| hotfix → main | Merge（打 Tag） | `git merge --no-ff hotfix/xxx` |
| hotfix → develop | Merge（回合修复） | `git merge hotfix/xxx` |
| develop → main（Trunk模式） | Squash and Merge | `git merge --squash feature/xxx` |

> **禁止**：直接在 `main` 或 `develop` 上提交代码。所有变更必须通过合并请求（Pull Request / Merge Request）。

## 4. 代码合并请求（Pull Request）规范

### 4.1 PR 创建标准

| 检查项 | 要求 |
|--------|------|
| 分支命名 | 符合命名规范 |
| 提交信息 | 符合提交信息格式 |
| 编译 | 本地编译通过 |
| 单元测试 | 新增代码配套单元测试，所有测试通过 |
| 代码规范 | 通过 Lint / Checkstyle 检查 |
| 冲突 | 已解决与目标分支的冲突 |

### 4.2 PR 描述模板

```markdown
## 变更内容
[简要描述本次变更的内容和目的]

## 关联需求/缺陷
- [PROJ-123](链接) 增加登录页面

## 变更类型
- [ ] 新功能
- [ ] 缺陷修复
- [ ] 重构优化
- [ ] 文档更新
- [ ] 配置变更

## 测试范围
- [ ] 单元测试已通过
- [ ] 集成测试已通过
- [ ] 手动测试已验证

## 影响评估
[本次变更对现有功能、性能、安全的影响说明]

## 检查清单
- [ ] 代码符合编码规范
- [ ] 无未处理的 TODO / FIXME
- [ ] 已更新相关文档
- [ ] 数据库变更已有迁移脚本
- [ ] 已添加必要的日志和监控
```

### 4.3 合并条件

- [ ] 至少 [1] 位 Reviewer 批准
- [ ] 所有自动化检查通过（CI / Lint / Test）
- [ ] 无未解决的对话/评论
- [ ] 分支已更新到目标分支最新版本

# 第三部分：提交信息规范

## 5. 提交信息格式

### 5.1 推荐：Conventional Commits

```
<类型>(<作用域>): <描述>

[可选的详细说明]

[可选的关闭 Issue 引用]
```

**类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(auth): add OAuth2 login support` |
| `fix` | 缺陷修复 | `fix(order): fix NPE when order is null` |
| `refactor` | 重构 | `refactor(api): extract common validator` |
| `docs` | 文档 | `docs(readme): update quick start` |
| `style` | 代码格式 | `style(eslint): fix indentation` |
| `test` | 测试 | `test(payment): add unit tests` |
| `chore` | 构建/工具 | `chore(deps): upgrade spring to 3.2` |
| `perf` | 性能优化 | `perf(cache): reduce redis calls` |

### 5.2 提交粒度原则

- **一个提交只做一个逻辑变更**：不要混合功能添加和缺陷修复
- **提交可独立构建通过**：不允许提交中间编译失败的代码
- **提交信息解释"为什么"而非"是什么"**：代码本身说明"是什么"，提交信息说明"为什么这么改"

# 第四部分：构建与CI/CD规约

## 6. 构建规范

### 6.1 构建工具

| 语言/平台 | 推荐构建工具 |
|----------|------------|
| [Java/Kotlin] | [Maven / Gradle] |
| [JavaScript/TypeScript] | [npm / pnpm / yarn] |
| [Python] | [pip / poetry / uv] |
| [Go] | [Go modules] |
| [容器化] | [Docker / Docker Compose] |

### 6.2 构建要求

- 构建脚本必须可重复执行，不依赖本地环境
- 版本号统一管理，遵循 [SemVer](https://semver.org/) 规范
- 构建产物不可手动修改，必须通过 CI 生成
- 每次合并到 `main` 自动生成带版本号的构建产物

### 6.3 版本号规范（SemVer 2.0）

```
MAJOR.MINOR.PATCH[-预发布号+构建元数据]

MAJOR：不兼容的 API 修改
MINOR：向下兼容的功能新增
PATCH：向下兼容的问题修正
预发布号：alpha / beta / rc
示例：v1.2.0、v2.0.0-beta.1、v1.3.0-rc.2
```

## 7. CI/CD 流水线规范

### 7.1 流水线阶段

| 阶段 | 触发条件 | 内容 | 失败处理 |
|------|---------|------|---------|
| **提交检查** | 每次 Push | Lint → 单元测试 → 编译 | 阻止合并 |
| **合并检查** | PR 创建/更新 | 提交检查 + 集成测试 + 代码扫描 | 阻止合并 |
| **构建部署** | 合并到 `develop` | 构建 → 部署测试环境 → 冒烟测试 | 通知负责人 |
| **发布构建** | 创建 `release` 分支 | 构建 → 部署预发环境 → 全量自动化测试 | 阻断发布 |
| **生产部署** | 合并到 `main` | 构建 → 部署生产 → 健康检查 → 灰度验证 | 自动回滚 |

### 7.2 CI 检查项

| 检查项 | 工具 | 通过条件 |
|--------|------|---------|
| 代码规范检查 | [ESLint / Checkstyle / Ruff] | 0 error, 0 warning |
| 单元测试 | [Jest / JUnit / pytest] | 通过率 100%，覆盖率 ≥ [X]% |
| 安全扫描 | [SonarQube / Snyk] | 无 Critical/High 漏洞 |
| 依赖检查 | [Dependabot / Renovate] | 无已知 CVE |
| 构建验证 | [Maven / npm build] | 编译成功 |
| 镜像安全 | [Trivy / Clair] | 无高危漏洞 |
