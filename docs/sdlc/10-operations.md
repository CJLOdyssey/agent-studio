### 4. 告警规则

#### 4.1 告警级别定义

| 级别 | 名称 | 定义 | 响应要求 |
|------|------|------|----------|
| P0 | 紧急 | 核心功能完全不可用，或大面积用户受影响 | 立即响应，5 分钟内 |
| P1 | 严重 | 核心功能严重降级，或部分用户受影响 | 15 分钟内响应 |
| P2 | 一般 | 非核心功能异常，或有明显性能下降 | 30 分钟内响应 |
| P3 | 提示 | 潜在风险或轻微异常，需关注 | 1 小时内响应 |

#### 4.2 阈值模板

| 指标类型 | 告警条件 | 级别 | 持续时长 | 示例 |
|----------|----------|------|----------|------|
| CPU 使用率 | > [占位符]% | P2 | 5m | > 80% |
| 内存使用率 | > [占位符]% | P1 | 3m | > 90% |
| 磁盘使用率 | > [占位符]% | P1 | 5m | > 85% |
| HTTP 5xx 率 | > [占位符]% | P0 | 1m | > 1% |
| HTTP P99 延迟 | > [占位符]ms | P1 | 5m | > 2000ms |
| 错误日志突增 | 较上周期增长 [占位符] 倍 | P2 | 5m | 增长 5 倍 |
| 消息队列堆积 | > [占位符] 条 | P1 | 10m | > 10000 条 |
| 业务成功率 | < [占位符]% | P0 | 2m | < 95% |

#### 4.3 告警聚合与抑制规则

- **告警聚合**：相同服务、相同指标的告警在 [占位符] 分钟内合并为一条通知
- **告警抑制**：下游故障引起的上游告警，优先发送根因告警，抑制派生告警
- **静默窗口**：计划内变更期间，对受影响指标启用 [占位符] 分钟静默期
- **通知去重**：同一告警在 [占位符] 分钟内仅发送一次，状态未恢复不再重复通知

## Part B  应急响应 SOP

### 6. 应急响应流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  发现   │ → │  响应   │ → │  止损   │ → │  定位   │ → │  修复   │ → │  复盘   │
│ Discover│    │ Respond │    │ Mitigate│    │ Locate  │    │ Fix     │    │ Review  │
└────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘
     │              │              │              │              │              │
  监控告警      值班人员        降级/限流      日志/链路      代码/配置      复盘会议
  用户反馈      响应并确认      扩容/切换      指标分析      回滚/热修      报告输出
  主动巡检      组建应急群      熔断/降级      复现验证      验证恢复      改进跟踪
```

#### 6.1 发现（Discover）

| 发现渠道 | 责任人 | 处理要求 |
|----------|--------|----------|
| 监控告警自动触发 | 值班工程师 | 收到告警后 [占位符] 分钟内确认 |
| 用户反馈（客服/社交媒体） | 客服/值班工程师 | 记录并转交值班工程师，[占位符] 分钟内响应 |
| 主动巡检发现 | 巡检负责人 | 立即记录并评估影响范围 |

#### 6.2 响应（Respond）

1. 值班工程师收到告警后，在 [占位符] 分钟内点击"确认告警"
2. 初步判断故障级别（P0-P3）
3. 若判断为 P0 或 P1，立即通过 [占位符] 渠道通知技术负责人
4. 组建应急群组（飞书/钉钉/微信群：[占位符]），拉上相关开发人员
5. 指定事故指挥官（Incident Commander），统一指挥后续操作

#### 6.3 止损（Mitigate）

| 止损手段 | 适用场景 | 执行人 | 预计生效时间 |
|----------|----------|--------|--------------|
| 服务降级 | 非核心功能拖垮核心链路 | 值班工程师 | [占位符] 分钟 |
| 限流熔断 | 流量突增导致服务雪崩 | 值班工程师 | [占位符] 分钟 |
| 扩容 / 弹性伸缩 | 资源不足导致性能下降 | 运维工程师 | [占位符] 分钟 |
| 流量切换 | 单点故障，需切至备用集群 | 运维工程师 | [占位符] 分钟 |
| 配置回滚 | 配置变更引发故障 | 值班工程师 | [占位符] 分钟 |
| 代码回滚 | 版本发布引入 bug | 值班工程师 | [占位符] 分钟 |
| 数据修复 | 数据异常导致业务错误 | DBA / 开发工程师 | [占位符] 分钟 |

#### 6.4 定位（Locate）

1. 查看监控 Dashboard，确认故障范围和影响指标
2. 查询全链路追踪，定位异常服务或接口
3. 检索错误日志，查找异常堆栈和错误码
4. 对比故障前后变更记录（代码发布、配置变更、基础设施变更）
5. 必要时在测试环境复现问题
6. 记录排查过程中的关键发现和时间节点

#### 6.5 修复（Fix）

1. 根据定位结果，选择修复方案：
   - 配置修复：修改配置并推送，验证生效
   - 代码热修：通过 [占位符] 方式发布补丁
   - 版本回滚：回退至上一稳定版本
   - 数据修复：执行 DBA 审核后的修复脚本
2. 修复后持续观察 [占位符] 分钟，确认指标恢复正常
3. 恢复被降级的功能，解除限流/熔断
4. 通知客服团队故障已恢复，准备用户安抚话术

#### 6.6 复盘（Review）

1. 故障结束后 [占位符] 小时内组织复盘会议
2. 输出故障复盘报告（见第 9 节模板）
3. 制定改进措施，明确责任人和完成时间
4. 将改进措施录入工单系统，定期跟踪闭环
