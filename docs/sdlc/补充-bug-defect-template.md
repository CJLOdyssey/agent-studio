# 补充模板：Bug / Defect 记录模板

> **补充说明**：本模板填补 SDLC 文档体系中产品缺陷记录模板的空白。与 `13-quality-assurance.md` Part C（QA 过程问题跟踪）的区别在于：QA 问题跟踪表记录的是**流程/过程不符合项**（如评审未做、文档缺失），本模板记录的是**产品代码缺陷**（如功能异常、UI 错误、性能问题）。
>
> 适用工具：Jira、GitHub Issues、GitLab Issues 或 Markdown 文件管理。

---

## 1. 模板说明

每个缺陷记录应包含以下两部分：

- **Part A — 缺陷信息表**：创建时必填，用于缺陷登记与状态跟踪
- **Part B — 技术分析**（可选）：修复后补充，用于技术复盘与知识沉淀

---

## Part A — 缺陷信息表

### 1.1 基础信息

| 字段 | 说明 | 填写示例 |
|------|------|---------|
| 缺陷编号 | 唯一标识，格式：BUG-YYYYMMDD-XXX | BUG-20260630-001 |
| 标题 | 简短描述，格式：[模块] 问题概述 | [消息面板] regenerate 后切换版本原始 thinking 丢失 |
| 严重程度 | 致命 / 严重 / 一般 / 轻微 | 严重 |
| 优先级 | P0（立即处理）/ P1（本迭代）/ P2（下迭代）/ P3（待评估） | P1 |
| 状态 | 新建 / 已确认 / 处理中 / 已修复 / 已验证 / 已关闭 / 重新打开 / 已拒绝 | 新建 |
| 发现版本 | 发现缺陷时的版本号或 commit hash | v1.2.0 / a3f2e1c |
| 修复版本 | 修复该缺陷的版本号或 commit hash | v1.2.1 / b4d5f6a |
| 发现日期 | 发现缺陷的时间 | 2026-06-30 |
| 报告人 | 发现缺陷的人员 | Sisyphus |
| 责任人 | 负责修复的开发人员 | Sisyphus |
| 验证人 | 验证修复结果的人员 | Sisyphus |
| 相关模块 | 涉及的模块/文件路径 | `chatStore.ts`、`MessagesPanel.tsx` |

### 1.2 缺陷描述

| 字段 | 说明 | 填写示例 |
|------|------|---------|
| 环境 | 操作系统、浏览器、设备等 | Chrome 120 / macOS 14.3 |
| 前置条件 | 进入该缺陷状态所需的前提条件 | 对话已有一轮完整的 AI 回复（含 thinking） |
| 复现步骤 | 精确的复现步骤，一步一步描述 | 1. 发送消息等待 AI 回复<br>2. 点击"重新生成"按钮<br>3. 等待新版本生成完成<br>4. 点击"Previous version"切换回 v0 |
| 预期结果 | 应该发生的正确行为 | thinking 区域显示完整的原始思考内容 |
| 实际结果 | 实际发生的错误行为 | thinking 区域只显示"我们"，内容被截断 |
| 截图/日志 | 辅助定位的截图、录屏、控制台报错 | `currentThinkingVersions: []`（应为原始 thinking 文本） |

### 1.3 跟踪信息

| 字段 | 说明 | 填写示例 |
|------|------|---------|
| 关联需求/任务 | 关联的需求编号或 Task ID | REQ-20260601-001 |
| 关联 PR | 修复该缺陷的 Pull Request 链接 | [#123](https://github.com/...) |
| 标签 | 自定义分类标签 | `bug`, `frontend`, `zustand`, `thinking` |
| 备注 | 任何其他需要记录的信息 | 根因是 JS 空数组 `[]` 的 truthiness 问题 |

## Part B — 技术分析（修复后补充）

### 2.1 根因分析

**简要概括缺陷产生的根本原因**（1-2 句话）：

> 示例：`agentMsg.thinkingVersions` 为空数组 `[]`（非 `undefined`），`[] || fallback` 返回 `[]`，导致原始 thinking 被丢弃。

**数据流/调用链追踪**（必要时可配流程图）：

```
步骤 1：... → 步骤 2：... → 步骤 3：... → 缺陷表现
```

### 2.2 修复方案

**修改的文件和关键变更**：

| 文件 | 行号 | 修改内容 |
|------|------|---------|
| `frontend/src/stores/chatStore.ts` | ~496 | `xxx.thinkingVersions \|\| fallback` → `xxx.thinkingVersions?.length ? xxx.thinkingVersions : fallback` |

**关键代码 diff**（可选）：

```diff
- const a = b || fallback;
+ const a = b?.length ? b : fallback;
```

### 2.3 验证方法

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|---------|---------|
| 1 | 复现原始缺陷步骤 | 修复前应有问题 | 复现成功 |
| 2 | 应用修复后重复步骤 | thinking 完整显示 | ✅ 通过 |
| 3 | 检查 LSP / 编译 | 无新错误 | 0 errors |

### 2.4 预防措施

**如何避免同类问题再次发生**：

> 示例：在代码审查中关注 `||` 在数组/对象上的使用是否正确；对 Zustand store 中所有同类 `truthy array` 模式做一次排查。

## 2. 状态流转说明

```
[新建] → [已确认] → [处理中] → [已修复] → [已验证] → [已关闭]
   ↓          ↓          ↓           ↓
[已拒绝]  [重新打开]  [重新打开]  [重新打开]
```

| 状态 | 定义 |
|------|------|
| 新建 | 缺陷已登记，等待确认 |
| 已确认 | 开发/PM 确认是有效缺陷 |
| 处理中 | 正在修复 |
| 已修复 | 代码已修改，等待验证 |
| 已验证 | QA 验证通过 |
| 已关闭 | 缺陷处理完成，流程终结 |
| 重新打开 | 修复不彻底或问题复发 |
| 已拒绝 | 经评估非缺陷或无法复现 |

---

## 3. 使用建议

- **小型项目**：Part A + 状态流转 即可，Part B 可选
- **中大型项目**：建议 Part A+B 都填写，Part B 可作为团队知识沉淀
- **与 QA 问题跟踪表协同**：
  - QA 跟踪表 → 记录过程审计发现的不符合项（如"没有做代码审查"）
  - 本模板 → 记录产品代码中的具体缺陷（如"按钮点击无响应"）
